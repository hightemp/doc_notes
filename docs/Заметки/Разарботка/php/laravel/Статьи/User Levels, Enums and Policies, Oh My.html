Users are not equal.

You heard me. Some users are above others. They are allowed to do more than other users. They can see things other users cannot see. Sometimes they can even remove lower-class users!

Now, this is not a weirdly abstract horror movie script, nor is it a manifesto to rally the common users into guillotining the higher classes. It's simply the way apps are built, and there's really not much wrong with it. We're going to be doing the exact same today, actually. And by "we're doing" I mean that I'll show you how to do it, and you just sit back, grab a cup of coffee or tea, and read on. Let's go!

# The Goal

We'll set up an app where users can have 3 levels: Administrator, Contributor and Member. For that, we'll use the newly introduced Enum class that came with PHP 8.1. After that we'll set up some routes to edit user levels and delete other users, and finally we'll use policies to divide our users into 'cans' and 'cannots'.

Here are the steps in more detail:

  • Create a Users page where users are listed
  • Add a level to our User model
  • Make it possible to edit User levels
  • Use the User level to allow or deny access to functionality using a Policy
  • Make it possible to delete Users

Along the way, we'll learn about PHP's new Enums, model casting and authorization using Laravel's policies.

# Leveling Up the Users

Next let's look at how we can expand the User model to add levels to it. These will be defined ahead of time and will have limited options. Freely translated: the perfect breeding ground for an enum! More specifically aÂ BackedEnum. These have been added since PHP 8.1 so make sure your php version supports enums! Create a newÂ app/EnumsÂ directory and add the following in theÂ UserLevel.phpÂ file:

<?php

namespace App\Enums;

enum UserLevel: int
{
    case Member = 0;
    case Contributor = 1;
    case Administrator = 2;
}

In this case, we have 3 different levels: Member, Contributor and Administrator. These correspond to an integer, which is why it's called a Backed Enum. This way, we can use the human-readable name for each value when we're coding but only the integer value will be saved into the database. Neat!

Now, let's pop the hood of our User Model and tack on the user level. InÂ app/Models/User:

+ use App\Enums\UserLevel;

  ...

  protected $fillable = [
      'name',
      'email',
      'password',
+     'level',
    ];

...

  protected $casts = [
      'email_verified_at' => 'datetime',
+     'level' => UserLevel::class,
  ];

By adding theÂ levelÂ attribute to theÂ castsÂ array, we inform Laravel that we want the integer from the database to be cast to the enum with all its functionality. If we didn't add it, ourÂ levelÂ attribute would be an integer on our Model.

There are many days when I would continue coding on other features or jump into the frontend stuff, and completely forget to add the migration. Today is not that day.

Create the migration withÂ php artisan make:migration add_level_to_usersÂ and add the following:

We've added a column in theÂ usersÂ table, with anÂ integerÂ data type. The default value is 0, but there's no easy way to know what actual level that corresponds toâ€¦ But that's exactly why we're using enums, so let's change that line:

# Protecting the Users

Right now, every user can change the user level of every other user. That's not exactly what we want, is it?

We only wantÂ someÂ users to be able to change the user level of another user. And as luck might have it we've already divided up our users in 3 categories: Administrators, Contributors and Members. So let's say only Administrators can change user levels.

For that we'll need a Policy: RunÂ php artisan make:policy UserPolicyÂ to make one. We'll use Controller Helpers to validate our edit and update methods, like this inÂ UserLevelController:

public function edit(User $user)
    {
+       $this->authorize('updateLevel', $user);
        return view('userlevels.edit')->with('user', $user);
    }

    public function update(Request $request, User $user)
    {
+       $this->authorize('updateLevel', $user);

        $validated = $request->validate([
            'level' => ['required', new Enum(UserLevel::class)]
            ]);

        $user->level = $validated['level'];
        $user->save();

        return redirect(route('users.index'));
    }

So, how does this work? TheÂ $this->authorizeÂ checks if the logged in User has permission toÂ updateLevelÂ theÂ $userÂ .

Behind the scenes, Laravel will look for a Policy that can be used for theÂ $userÂ Model. It will look in theÂ app/PoliciesÂ andÂ app/Models/PoliciesÂ folders if it can find a Policy with the correct name:Â [name of Model] + Policy.phpÂ . In our case that will beÂ UserPolicy, which is exactly what we chose. This means that Laravel will automatically link our UserPolicy to actions we want to do with the User model.

In the UserPolicy, Laravel will run theÂ updateLevelÂ method with two parameters: the logged in User and the Model that user is trying to change. In our case this will be another User. IfÂ trueÂ is returned in the method then the action is allowed, else it will be forbidden. Let's quickly test out ourÂ updateLevelÂ method, add the following inÂ UserPolicy:

/**
 * @param User $loggedInUser the user that's trying to update the level of $model
 * @param User $model the user whose level is being updated by the $loggedInUser
 * @return bool
 */
public function updateLevel(User $loggedInUser, User $model)
{
  return false;
}

We're using a new method in the User model that doesn't exist yet:Â getNumberOfAdmins. Here's how it looks:

// Add this in the User model:

{

  return User::where('level', UserLevel::Administrator)->count();
}

There. Now, when an admin is logged in they can edit the user level of everyone, with one exception: they can't edit their own level if they are the only admin. Looks good!
